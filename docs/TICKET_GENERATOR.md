# ServiceNow / Jira-ready Change Ticket Summaries

## Overview

PreflightOps can turn a risk assessment into a copy/paste-ready Markdown change
summary. The summary is structured like a change record — change title, service,
environment, business impact, risk level and score, approvals, rollback /
monitoring / validation plans, risk findings, and recommended actions — so you
can paste it straight into ServiceNow, Jira, CAB notes, a change advisory
review, or any internal approval workflow.

Key points:

- It works **without** ServiceNow or Jira credentials.
- It requires **no network access** — the summary is generated locally.
- It is useful for formal change governance and audit trails.
- An optional **live API push** to ServiceNow or Jira exists, but it is strictly
  opt-in and must be explicitly enabled (see below).

## Offline by default

> By default, PreflightOps makes no outbound calls. It only reads local input
> files and writes local reports. Live ServiceNow or Jira pushes only happen when
> `--servicenow` or `--jira` is explicitly provided.

## Generate a Markdown ticket summary

Add `--ticket-output` to write the change summary alongside the normal report:

```bash
preflightops \
  --services examples/services-critical-risk.yaml \
  --change examples/change-critical-risk.yaml \
  --terraform examples/terraform-critical.txt \
  --k8s examples/k8s-risk.yaml \
  --output report.md \
  --json-output report.json \
  --ticket-output ticket.md
```

`--ticket-output` writes a Markdown change summary to the given path. It makes no
network calls and needs no credentials. A full sample is checked in at
[`examples/ticket-critical.md`](../examples/ticket-critical.md).

## Example output

A `CRITICAL` production change produces a summary like this (abbreviated):

```markdown
# Production Change Summary

## Change Title
Migrate payments-core to new IAM roles and database

## Service
payments-core

## Environment
production

## Business Impact
Customers cannot be charged and revenue stops immediately.

## Risk Level
CRITICAL

## Risk Score
100/100

## Required Approvals
Deployment should not proceed until critical gaps are resolved and senior approval is obtained.

## Rollback Plan
No rollback plan detected.

## Monitoring Plan
No monitoring plan detected.

## Validation Plan
No validation plan detected.

## Recommended Deployment Window
Do not deploy until critical gaps are resolved. If deployment is unavoidable, use an approved emergency change process.

## Risk Findings
- Production change has no valid rollback plan
- No monitoring plan defined
- Post-deploy validation plan is missing
- IAM role change detected
- Deployment does not define readinessProbe

## Recommended Actions
- Provide a valid, tested rollback plan.
- Define a monitoring plan (dashboards, alerts, success criteria).
- Add a post-deploy validation plan.
- Deployment should be blocked until rollback, monitoring, ownership, or validation gaps are resolved.

## Notes
Generated by PreflightOps as a copy/paste-ready change summary for ServiceNow, Jira, CAB review, or internal approval workflows.
```

The approval wording and the recommended deployment window vary by risk level.
When the change request omits a rollback, monitoring, or validation plan, the
corresponding section falls back to a clear "No … plan detected." line so the gap
is visible to reviewers.

## Custom ticket templates

The change summary uses a built-in layout, but you can tailor it with
`--ticket-template` — a YAML or JSON file. Templates need no extra dependencies
(the file is parsed with the same YAML loader, which also accepts JSON).

```bash
preflightops \
  --services examples/services-critical-risk.yaml \
  --change examples/change-critical-risk.yaml \
  --ticket-output ticket.md \
  --ticket-template my-cab-template.yaml
```

A template is a mapping with any of these keys; anything you omit falls back to
the built-in default, so partial overrides are fine:

| Key | Effect |
| --- | --- |
| `title` | The `# H1` document title. |
| `sections` | An ordered list of `{ "heading": <text>, "content": <key> }`. Replaces the default section order entirely. |
| `approvals` | Per-risk-level approval wording (`LOW` / `MEDIUM` / `HIGH` / `CRITICAL`). Merged over the defaults. |
| `deployment_windows` | Per-risk-level deployment-window wording. Merged over the defaults. |
| `notes` | The footer text. |

Each `sections[].content` must be one of the available fields:
`change_title`, `service`, `environment`, `business_impact`, `risk_level`,
`risk_score`, `required_approvals`, `rollback_plan`, `monitoring_plan`,
`validation_plan`, `deployment_window`, `risk_findings`, `recommended_actions`,
`notes`. An unknown `content` key (or an empty `sections` list) is rejected with
a clear error, so a malformed template fails fast instead of producing a broken
ticket.

The template shapes the summary everywhere it is produced — both the
`--ticket-output` file and any opt-in ServiceNow / Jira push — so the live record
matches the offline summary.

Example `my-cab-template.yaml`:

```yaml
title: Internal CAB Change Record
sections:
  - heading: Summary
    content: change_title
  - heading: Blast Radius
    content: service
  - heading: Risk
    content: risk_level
  - heading: Sign-off
    content: required_approvals
approvals:
  CRITICAL: CAB chair must approve in person before deployment.
notes: Filed via the internal CAB process.
```

## Optional ServiceNow push

`--servicenow <INSTANCE_URL>` creates or updates a real ServiceNow
`change_request` from the generated summary. It is the live-API counterpart to
`--ticket-output` and is **optional** — nothing reaches the network unless you
pass this flag.

- **What it does:** posts the change summary as a `change_request` via the
  ServiceNow Table API. A deterministic correlation id is stored on the record,
  so a second run for the same change updates the existing record instead of
  creating a duplicate.
- **Required environment variables:** `SERVICENOW_USER` and
  `SERVICENOW_PASSWORD`.
- **Credentials are read from the environment only** — never passed inline on the
  command line and never hard-coded. The instance URL is the only argument.
- **Confirmation:** before any push, PreflightOps prints the target URL and the
  correlation id and asks for an explicit `y/N` confirmation. Pass `--yes`
  (alias `--assume-yes`) to skip the prompt in CI. When there is no interactive
  terminal and `--yes` was not given, the push is skipped (no API call).

```bash
export SERVICENOW_USER="..."
export SERVICENOW_PASSWORD="..."

preflightops \
  --services examples/services-critical-risk.yaml \
  --change examples/change-critical-risk.yaml \
  --ticket-output ticket.md \
  --servicenow https://example.service-now.com
```

## Optional Jira push

`--jira <BASE_URL>` creates or updates a Jira issue from the same summary. It is
**optional** and makes no network call unless the flag is provided.

- **What it does:** creates a Jira issue (or updates the matching one, found by a
  correlation-id label) with the change summary as the description.
- **Required environment variables:** `JIRA_EMAIL`, `JIRA_API_TOKEN`, and
  `JIRA_PROJECT_KEY`. `JIRA_ISSUE_TYPE` is optional and defaults to `Task`.
- **Credentials are read from the environment only** — never inline, never
  hard-coded. The base URL is the only argument.
- **Confirmation:** same as ServiceNow — an explicit prompt unless `--yes` /
  `--assume-yes` is given; skipped with no API call when there is no terminal and
  `--yes` was not passed.

```bash
export JIRA_EMAIL="..."
export JIRA_API_TOKEN="..."
export JIRA_PROJECT_KEY="OPS"

preflightops \
  --services examples/services-critical-risk.yaml \
  --change examples/change-critical-risk.yaml \
  --ticket-output ticket.md \
  --jira https://example.atlassian.net
```

## Safety notes

- Do not include secrets in your service catalog or change files.
- Review the generated ticket before submitting it to a change-management system.
- Do not push to ServiceNow/Jira from untrusted pull requests.
- Prefer offline ticket generation (`--ticket-output` only) for first adoption.
- Use live push (`--servicenow` / `--jira`) only from trusted CI contexts, with
  credentials supplied through your CI secret store.

## What this is not

- Not a full ServiceNow replacement.
- Not a full Jira workflow engine.
- Not a CAB replacement.
- Not a security scanner.
- Not an incident management system.

## Recommended rollout

1. Start with `--ticket-output` only.
2. Review the generated tickets manually.
3. Customize templates (`--ticket-template`) if your change format needs it.
4. Enable live ServiceNow/Jira push (`--servicenow` / `--jira`) only in trusted
   workflows.
5. Later, integrate with your team's governance processes (CAB, approvals).
